# Decision Log — 2026-07-20

## Member Chat System

Added a real-time member chat system with direct and group messaging.

## Problem

Members need to communicate with each other through the platform via both one-on-one direct messages and group chat rooms.

## Solution

Four-table chat system:
- **chat_rooms** (topic): stores room name, type (direct/group), creator
- **chat_messages** (content): stores message text, image/file paths per room
- **chat_room_members** (mapping): many-to-many between members and rooms
- **read_receipts** (read status): per-message read tracking per member

## Key Decisions

1. **Room type**: `chat_rooms.type` uses CHECK constraint (`'direct'` or `'group'`). Direct rooms allow NULL name (dynamically derived from the other participant's name). Group rooms require a name.

2. **Read receipts**: Separate table `read_receipts` with composite PK (message_id, member_id) for per-message read tracking, rather than a simple `last_read_at` on the member-room mapping.

3. **Media support**: `chat_messages.image_path` and `file_path` columns (VARCHAR(500), default empty string) to support text + image/file uploads, following the same pattern as `announcements.image_path`.

4. **SQL functions**: All CRUD goes through PostgreSQL functions (`create_chat_room`, `send_message`, `list_messages`, etc.) following the established pattern from migrations 008 and 010.

5. **Route design**: Follow existing POST-based convention.
   - Nested under `/api/chat/rooms/{roomId}/messages/{messageId}`
   - Mutations use POST (create, update, delete)
   - Reads are public GET (matching existing pattern)
   - Mutations require auth middleware

6. **Pagination**: Messages use cursor-based pagination (`before_id` query param). Room listing returns latest message preview and unread count.

7. **Implementation order**:
   - Migration → Domain structs → Errors → Interface → Repository → Handler → Routes → Wiring → Tests
