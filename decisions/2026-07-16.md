# Decision Log — 2026-07-16

## Three-Tier Product Schema

Decided to split product data into 3 tables:
- **products** — main info (type, name, status)
- **product_details** — info sections (return policy, usage, specs, shipping)
- **product_prices** — price variants (adult/child ticket, size variants)

## Key Decisions

1. **Three-layer structure**: prices are nested under details, not directly under products.
   - `products 1→N product_details 1→N product_prices`
   - Prices belong to a specific detail section.

2. **Detail types**: use fixed enum values — `return_policy`, `usage_instructions`, `specifications`, `shipping_info`.

3. **Multi-currency support**: `product_prices` has a `currency` column (default `TWD`).

4. **Cascade delete**: deleting a product cascades to its details and prices.

5. **API design**: follow existing POST-based convention for all mutations.

6. **Implementation order**:
   - SQL migration → domain structs → repo interface → in-memory repo → handlers → routes → tests
