# Decision Log — 2026-07-18

## Inventory System

Decided to add an inventory tracking system for product price variants.

## Problem

Products with multiple price tiers (e.g., adult/child tickets) need individual inventory tracking per price variant. Each price tier can have its own stock pool and track individual items.

## Solution

Two-table inventory system:
- **inventories** (summary): one per `product_price`, tracks total_quantity, sold_quantity, name, status
- **inventory_items** (detail): individual items under an inventory, each with item_code, status, cost, date tracking

## Key Decisions

1. **inventories ↔ product_prices**: one-to-one (UNIQUE constraint on product_price_id). Each price variant has exactly one inventory pool.

2. **inventories ↔ inventory_items**: one-to-many. An inventory has many individual items (e.g., individual tickets in a ticket pool).

3. **Status fields**:
   - Inventory status: `銷售中`, `完售`, `註銷`
   - Item status: `可用`, `出售`, `註銷`

4. **Route design**: follow existing POST-based convention.
   - Nested under `/api/inventories/{id}/items/{iid}`
   - Mutations use POST (create, update, delete)
   - Reads are public GET (matching existing pattern)
   - Mutations require auth middleware

5. **Cost tracking**: each inventory_item has a `cost` field (NUMERIC) for unit cost tracking.

6. **Date tracking**: `date_added` (DATE) tracks when an item was added; `status_updated_at` tracks when item status last changed.

7. **Implementation order**:
   - Migration → Domain structs → Errors → Interface → Repository → Handler → Routes → Wiring → Tests

- Removed `name`, `total_quantity`, `sold_quantity` columns from `inventories` table
- `name` is now computed dynamically via `CONCAT(p.name, '-', pr.label)` in query
- `total_quantity` and `sold_quantity` are computed via `COUNT(it.id)` with FILTER
- Updated domain structs, repository SQL, handler logic, and seed script accordingly

## PostgreSQL Functions (migration 008)

- Moved all raw SQL from Go repository files into 27 PostgreSQL functions
- Application now calls `SELECT * FROM function_name($1, $2, ...)` instead of inline SQL
- Update/Delete operations consolidated into single function calls with RETURNING (no separate SELECT for timestamps)
- Delete functions use PL/pgSQL `RETURN FOUND` for reliable boolean return
- Functions organized by domain: members (4), products (6), product_details (3), product_prices (4), inventories (6), inventory_items (4)
