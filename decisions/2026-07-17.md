## Product Detail Schema Redesign

Decided to replace the generic `detail_type` + `title` + `content` + `sort_order` pattern with fixed, named columns for product details.

## Problem

The original `detail_type` enum (`return_policy`, `usage_instructions`, `specifications`, `shipping_info`) was too generic and didn't match actual product requirements. The three fields needed are all textual descriptions that don't benefit from a key-value pattern.

## Solution

Replace the four dynamic columns with three fixed TEXT columns:
- `introduction` — 產品介紹
- `usage_instructions` — 使用說明
- `return_policy` — 退貨說明

## Key Decisions

1. **Fixed columns over enum**: product_details now has explicit `introduction`, `usage_instructions`, `return_policy` columns. No more `detail_type` + `title` + `content` + `sort_order`.

2. **One detail record per product**: since there's no longer a need for multiple detail rows with different types, each product has exactly one `product_details` row.

3. **product_prices unchanged**: prices still FK to `product_details(id)`, unchanged.

4. **Backward compatibility**: existing databases use `004_alter_product_details.sql` migration. Fresh installs get the new schema from the updated `001_create_products.sql`.

5. **Migration applied to both databases**: `productdb` and `productdb_test` both altered via the same migration SQL.

## Session Rotation

Decided to implement Session Rotation (session_key regeneration) to mitigate session hijacking.

## Problem

The `session_key` cookie is the only credential for API authentication. If an attacker steals it (via XSS, network sniffing, or browser extension), they can impersonate the victim indefinitely (up to the 24h TTL).

## Solution: Session Rotation

Every authenticated request generates a new `session_key` and invalidates the old one. This limits the window of a stolen session key to at most one request.

## Key Decisions

1. **Interface**: `SessionRepository` gets a new `Rotate(ctx, oldKey string) (*Session, error)` method. It atomically deletes the old session and creates a new one with the same `member_id` and expiry, but a fresh `session_key`.

2. **Rotation in middleware, not handlers**: The auth middleware calls `Rotate` after validation and before the handler runs. If rotation succeeds, the response includes a `Set-Cookie` header with the new key. Handlers are unaware of rotation.

3. **No rotation on public endpoints**: Only routes behind `AuthMiddleware` trigger rotation. Login creates the initial session; public endpoints (list products, register) do not rotate.

4. **Error handling**: If `Rotate` returns an error (session not found, expired), the middleware returns 401. The old cookie is invalidated.

5. **Cache impact**: Rotation adds one map write + one delete per authenticated request. Negligible for an in-memory cache.

6. **DB fallback**: `SessionRepositoryPGX` also implements `Rotate` for compatibility, using a database transaction.

7. **Implementation order**:
   - Add `Rotate` to `SessionRepository` interface
   - Implement in `SessionCache`
   - Implement in `SessionRepositoryPGX` (unused but consistent)
   - Update `AuthMiddleware` to use `Rotate`
   - Update tests

## Security Considerations

- Rotation alone does not prevent hijacking within the same request window. Combined with fingerprinting (IP + User-Agent hash), the attack window is reduced to near zero.
- Every rotation sends a new `Set-Cookie` header; this is transparent to the browser.

## Device Fingerprint

Decided to bind each session to a device fingerprint (`IP + User-Agent` hash) to prevent stolen cookies from being used on different devices.

## Problem

Session rotation only protects when the legitimate user is actively using the site (the stolen key is rotated away before the attacker can use it). If an attacker steals a cookie while the user is offline, they can use it indefinitely (up to the 24h TTL). Rotation actually helps the attacker by issuing a fresh key.

## Solution: Device Fingerprint

Every session stores `SHA256(IP + "|" + User-Agent)`. The auth middleware computes the fingerprint of the current request and passes it to `Rotate`. If it doesn't match the stored fingerprint, `Rotate` returns `ErrDeviceMismatch` and the request is rejected.

## Key Decisions

1. **Fingerprint algorithm**: `SHA256(IP + "|" + User-Agent)`. IP is extracted from `X-Forwarded-For` → `X-Real-Ip` → `RemoteAddr` (strip port), in priority order, to support reverse proxy deployments.

2. **Check in Rotate, not after**: The fingerprint check is inside `Rotate` (under the same lock), so the old session is atomically invalidated on mismatch. The attacker cannot retry with the same cookie.

3. **Fingerprint update on each request**: When a match succeeds, the rotated session stores the current request's fingerprint. This allows legitimate IP changes (e.g., mobile switching WiFi) while still blocking cross-device attacks.

4. **Empty fingerprint is backward-compatible**: If `DeviceFingerprint` is empty (old session created before this change), the check is skipped.

5. **Same-device tolerance**: IP changes are tolerated because the fingerprint includes User-Agent, which stays constant on the same device. An attacker would need the exact same IP + UA to pass — practically impossible.

6. **New helper, new error**: `DeviceFingerprint(r)` in `src/api/fingerprint.go`. `ErrDeviceMismatch` in `domain/errors.go`.

7. **Implementation order**:
   - Add `DeviceFingerprint` to `Session` domain struct
   - Add `ErrDeviceMismatch` error sentinel
   - Update `Rotate` interface to accept `fingerprint string`
   - Implement fingerprint check in `SessionCache.Rotate`
   - Add `DeviceFingerprint` helper function
   - Update `AuthMiddleware` to compute and pass fingerprint
   - Update `LoginMember` to set fingerprint on session creation
   - Add tests for fingerprint mismatch

## GET and Update APIs for Detail & Prices

Decided to add read and update endpoints for `product_details` and `product_prices`.

## Problem

The original change only added create endpoints for details and prices. There was no way to read or update existing data.

## Solution

Add 5 new endpoints:
- `GET /api/products/{id}/detail` — get the single detail for a product
- `POST /api/products/{id}/detail/update` — update detail (auth required)
- `GET /api/products/{id}/detail/prices` — list all prices for a product's detail
- `GET /api/products/{id}/detail/prices/{pid}` — get a single price
- `POST /api/products/{id}/detail/prices/{pid}/update` — update a price (auth required)

## Key Decisions

1. **Read endpoints are public**: matching `GetProduct`/`ListProducts` pattern, no auth required.
2. **Update endpoints require auth**: any authenticated user can update (no ownership check), matching the project's existing auth model.
3. **ListPrices resolves detail ID internally**: the handler fetches the detail by product ID, then lists prices by detail ID — the client only needs the product ID.
4. **Route design**:
   - Uses `/detail` (singular) for the single-detail-per-product concept, distinct from the existing `/details` (plural) create route.
   - Update follows the existing `POST /{resource}/update` convention.

## Add Status Field to Product

Decided to expose the `status` column on the `products` table in the API.

## Problem

The `status` column existed in the database (`VARCHAR(50) NOT NULL DEFAULT 'active'`) but was not exposed in the Go struct or API. There was no way to set or read a product's status.

## Solution

Add `Status` to `Product`, `CreateProductRequest`, and `UpdateProductRequest` structs.

## Key Decisions

1. **Default "active"**: If `status` is not provided in create/update requests, it defaults to `"active"` in the repository layer.
2. **NOT NULL handling**: Since the column is `NOT NULL`, the repository always passes a non-nil value — defaulting to `"active"` when empty.
3. **Full wiring**: Status is included in all SELECT queries, INSERT, and UPDATE statements.
